# 装箱装托规划系统 - 产品文档（V1.1 草案）

## 1. 产品定位
面向外贸/供应链打包环节，基于订单型号与数量，自动生成“内盒 -> 外箱 -> 托盘”的最优装载方案，目标是在不显著增加拼箱/拼托复杂度的前提下，尽可能减少外箱数与托盘数，并输出可执行的结果文件与三维可视化。

## 2. 业务目标与成功标准
### 2.1 业务目标
1. 将人工装箱/装托计划时间从小时级缩短到分钟级。
2. 降低外箱和托盘使用量，减少运输与仓储成本。
3. 输出可追溯、可复核、可视化的发货方案。

### 2.2 成功指标（建议）
1. 计算成功率 >= 99%（可输出至少一套可执行方案）。
2. 与人工基准比，外箱总数平均下降 >= 5%。
3. 与人工基准比，托盘总数平均下降 >= 5%。
4. 单次计算耗时 <= 60 秒（中等订单规模）。

## 3. 术语定义
1. 内盒：产品一级包装盒。
2. 外箱：内盒组合后的运输纸箱。
3. 托盘：外箱装托后的运输单元。
4. 拼箱：同一外箱中混装多型号产品。
5. 拼托：同一托盘中混装多规格外箱。
6. 样品箱：不追求紧密摆放、优先减少使用数量的特殊箱。

## 4. 用户角色与场景
1. 计划员：上传订单，生成方案，导出结果。
2. 复核员：检查拼箱/拼托合理性，确认发货。
3. 管理员：维护基础数据（装箱.xls、托盘/纸箱尺寸.xlsx）。

典型场景：
1. 导入订单 Excel。
2. 系统自动计算装箱与装托方案。
3. 用户查看多个候选方案并选择。
4. 导出标准 Excel + 生成三维网页链接用于沟通与复核。

## 5. 产品范围
### 5.1 本期（V1）
1. Excel 导入与校验。
2. 规则驱动的装箱与装托计算。
3. 多候选方案生成与选择。
4. Excel 标准模板导出。
5. 三维可视化（内盒、外箱、托盘三级联动）。

### 5.2 暂不包含
1. 与 ERP/WMS 直接双向同步。
2. 自动学习型参数调优（先用配置化规则）。
3. 移动端离线能力。

## 6. 输入与输出规范
### 6.1 输入
1. 订单 Excel：单位信息、订单编号、型号、数量。
2. 型号-内盒映射：`装箱.xls`（读取所有 sheet）。
3. 内盒-外箱-托盘映射：`托盘，纸盒纸箱尺寸.xlsx`。

输入特殊规则：
1. 订单文件 A1 使用固定文案：`合并` / `不合并`。
2. A1 = `不合并`：该单位下不同订单产品不可同箱。
3. A1 = `合并`：允许该单位下不同订单按规则拼箱。

### 6.2 输出
1. 装箱结果表：按订单展示外箱规格、拼箱关系、件数、重量。
2. 装托结果表：按托盘展示外箱明细、排列方式、每托重量、汇总。
3. 三维可视化链接：可旋转/缩放/拖动查看摆放结果。
4. 异常报告：缺失映射、超限重、无法装入等问题。

## 7. 约束与优化目标
### 7.1 约束（硬约束）
1. 默认内盒仅正放，特殊情况下可侧放（需满足空余与少量尾货条件）。
2. 托盘可用内部尺寸 = 外部尺寸扣减：长宽 -8cm，高 -13cm。
3. 托盘自重 30kg，限重 1250kg。
4. 默认优先同尺寸外箱同托，再考虑相近尺寸拼托。

### 7.2 优化目标（软约束）
1. 最小化托盘数。
2. 最小化外箱数。
3. 最小化拼箱/拼托复杂度。
4. 提高空间利用率与重量均衡。
5. 默认优先级固定为：托盘数 > 外箱数 > 复杂度。
6. 支持权重参数配置（用于结果不理想时调参），但不改变硬约束。

## 8. 功能需求
### 8.1 文件管理
1. 上传订单文件、规则文件。
2. 自动识别模板版本与字段映射。
3. 输出校验错误列表（行号+原因+修复建议）。

### 8.2 计算任务
1. 一键计算，默认输出 3 套候选方案。
2. 方案类型固定为：保守（少拼箱）/均衡/极致省箱。
3. 支持任务状态：排队、计算中、成功、失败。
4. 支持手工锁定规则：可指定型号必须独立装箱/独立装托。

### 8.3 方案展示
1. 二维表格：按订单、按外箱、按托盘切换视图。
2. 差异对比：候选方案在箱数/托数/重量上的差异。
3. 三维视图：托盘层级与外箱摆放方向可视化。

### 8.4 导出
1. 按固定模板导出 Excel（参考样例文件，V1 不做多模板）。
2. 导出包含“装箱明细 + 装托明细 + 汇总统计”。

## 9. 关键实现方案
### 9.1 Excel 解析与规则标准化
1. 使用统一解析层读取多 sheet，并映射为内部标准数据结构。
2. 对“内盒*数量”“排列方式(如5*5*2)”做语法解析器，转为结构化字段。
3. 引入规则版本号，支持后续模板变化兼容。

### 9.2 装箱求解引擎（核心）
1. 采用“规则优先 + 启发式优化”两阶段算法。
2. 阶段A（可行解生成）：严格按你给定的 3 级优先顺序遍历（同型号 -> 同内盒 -> 可升级内盒）。
3. 阶段B（优化）：在可行解基础上进行局部搜索（交换、合并、拆分）减少外箱数，并控制拼箱复杂度。
4. 输出多候选解：保守/均衡/省箱三档，供人工选择。

### 9.3 装托求解引擎（核心）
1. 以托盘可用内尺寸与限重为硬约束，先正放后竖放。
2. 先同规格聚类，再做相近规格拼托；104/105/111 作为常见拼托优先组合。
3. “相近规格外箱”按尺寸差阈值判定（阈值参数化，可配置）。
4. 对尾货执行“最后一托加大加高”策略：仅在规定范围内，且至少可减少 1 托时触发。
5. 每步记录决策日志，确保结果可解释。

### 9.4 三维可视化
1. 前端使用 WebGL（Three.js）渲染托盘、外箱、内盒。
2. 后端输出 `layout.json`（坐标、尺寸、朝向、层级、重量）。
3. 前端根据 `layout.json` 实时渲染，并支持筛选订单/托盘。

### 9.5 导出引擎
1. 采用模板填充方式生成 Excel，保证与现有格式一致。
2. 对拼箱场景做单元格合并逻辑（按外箱规格与分组）。
3. 自动附加汇总行：总箱数、总托数、总毛重、总件数。

## 10. 建议技术架构
1. 前端：Web（Vue/React 均可），负责上传、任务管理、结果展示与三维可视化。
2. 后端：Python（FastAPI）提供上传、计算、导出 API。
3. 计算内核：独立 `solver` 模块，便于离线测试与性能优化。
4. 存储：
- 关系库保存任务与结果元数据。
- 对象存储保存原始 Excel、导出文件、layout.json。
5. 异步任务：队列执行计算，避免阻塞接口。

## 11. 数据模型（核心实体）
1. `OrderLine`：单位、订单号、型号、数量、限制标记。
2. `ModelPackRule`：型号、默认内盒、单重、默认每箱数量、默认箱规等。
3. `BoxPalletRule`：内盒尺寸、外箱规格、排列方式、默认托盘规格等。
4. `PackingPlan`：外箱清单、拼箱关系、重量体积统计。
5. `PalletPlan`：托盘清单、摆放坐标、重量校验、汇总。

## 12. 非功能需求
1. 可追溯：每次计算保留输入快照、规则版本、决策日志。
2. 稳定性：异常可定位到具体订单行与规则项。
3. 性能：中等规模任务 60 秒内给出结果。
4. 可维护：规则配置化，避免硬编码。

## 13. 验收标准（V1）
1. 可正确读取并校验输入模板。
2. 可生成至少一套满足硬约束的装箱装托方案。
3. 导出格式与既有样例一致（关键列、合并、汇总正确）。
4. 三维页面可准确呈现托盘与外箱布局。
5. 输出结果包含异常与不可解提示。

## 14. 风险与应对
1. 基础数据缺失或冲突：增加预校验与缺省规则回退。
2. 订单规模大导致性能下降：分批求解 + 启发式剪枝 + 异步队列。
3. 规则频繁变化：规则版本化与配置中心管理。
4. 结果可解释性不足：输出步骤日志与命中规则明细。

## 15. 已确认决策
1. 多目标优先级固定为“托盘数 > 外箱数 > 复杂度”，并增加可配置权重输入能力。
2. A1 单元格使用固定文案：`合并` / `不合并`。
3. 候选方案默认输出 3 套：保守/均衡/极致省箱。
4. “相近规格外箱”采用尺寸差阈值判定。
5. “最后一托加大加高”仅在规定范围内且至少减少 1 托时触发。
6. 允许用户手工锁定某些型号必须独立装箱/独立装托。
7. 三维展示在 V1 展示到内盒层级。
8. 导出模板采用单一固定样式。
